# Workflow strategy

Behold our grand plan of making a PUTwiki.

## Work cycle

We meet weekly. We upload notes from every meeting on [AKAI's google drive](https://drive.google.com/drive/u/1/folders/1cH3tak3CIDcT1kPCZ9xzqBoHhE2RHrGA) (access restricted to club members).

### Meetings & decision making

- We hold brief planning meetings to review progress, blockers, and priorities.
- We discuss tasks in progress and confirm goals for the iteration.
- When needed we use Figma board for brainstorming and visual references.

### Reflecting decisions for next iteration

- Review issues in **Backlog** on GitHub Projects.
- Move selected issues to **Todo** for the current iteration.
- Ensure each issue has a clear description and acceptance criteria.

**NOTE**: We use issue templates (and PR template too). The most common we use daily are: **Task** and **User story**.

### Working on issues

1. Pick an unassigned issue from **Todo**.
2. Assign it to yourself and move it to **In Progress**.
3. Create a branch following [branching guidelines](/docs/workflow/git.md).
4. Complete the work and prepare a PR.

### Review & feedback

1. Open a PR following [PR lifecycle](/docs/workflow/git.md).
2. Move the issue to **In Review**.
3. Notify the team for review.
4. After merge, the issue automatically moves to **Done**.
